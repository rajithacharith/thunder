# Registration Flows

A registration flow defines the steps a new user completes when creating an account. Using the Flow Builder in the Thunder Console, you can design the sign-up experience — collecting user information, verifying identities, and configuring post-registration behavior such as auto-login or welcome notifications.

## Before You Begin

- Ensure you have at least one application configured in the Thunder Console.
- Familiarize yourself with [Flow Concepts](./flow-concepts) to understand node types and the left panel.

## Key Executors for Registration Flows

The following executors are most commonly used in registration flows:

| Executor | Description |
|----------|-------------|
| **Attribute Collector** | Collects user profile data as defined in the user schema. |
| **Provisioning** | Creates the user account in the store. |
| **User Consent** | Captures explicit user consent for terms, scopes, or policies. |
| **Auth Assertion Generator** | Issues an authentication assertion after successful registration (used when auto-login is enabled). |
| **Send SMS OTP** / **Verify SMS OTP** | Sends and verifies an OTP for phone or email verification. |

## END Node Settings for Registration

The END node in a registration flow includes settings that control what happens after the flow completes. Click the settings icon on the END node to configure:

| Setting | Description |
|---------|-------------|
| **Activate Account** | Activate the user account as soon as registration completes. Turn off if you require verification before activation. |
| **Auto-Login** | Automatically sign in the user after registration. When turned on, add an **Auth Assertion Generator** executor before the END node. |
| **Send Notification** | Send a welcome or confirmation notification to the new user. |

## Example — Basic Self-Registration

This flow collects a username, email, and password from a new user and provisions the account immediately.

### Canvas Structure

```
START → Registration Form (View) → Attribute Collector (Executor) → Provisioning (Executor) → END
```

### How to Set It Up

1. On the Flows page, click **+ Create New Flow**.
2. Enter a name such as **Self Registration** and set the type to **Registration**, then click **Create**.
3. In the left panel, expand **Steps** and click **+** next to **Blank View** to add a registration form.
4. Select the View node, then expand **Components** in the left panel and add:
   - **Text Input** — for the username.
   - **Email Input** — for the email address.
   - **Password Input** — for the password.
   - **Button** — to submit the form.
5. Expand **Executors** and add **Attribute Collector** to the canvas.
6. Connect the View node's output to the **Attribute Collector** executor.
7. Add a **Provisioning** executor and connect the Attribute Collector success path to it.
8. Add an **END** node and connect the Provisioning success path to it.
9. Connect each executor's failure path to an error View.
10. Click the settings icon on the END node and turn on **Activate Account** and **Send Notification**.
11. Click **Save**.

## Example — Registration with Email Verification

This flow requires the user to verify their email address before their account becomes active.

### Canvas Structure

```
START → Registration Form (View) → Attribute Collector (Executor) → Provisioning (Executor)
          → Send OTP (Executor) → OTP Input (View) → Verify OTP (Executor) → END
```

### How to Set It Up

1. Create a new **Registration** flow.
2. Build the registration form steps (View + Attribute Collector + Provisioning) as described above.
3. After the **Provisioning** executor's success path, add a **Send SMS OTP** executor from the Executors panel.
4. In **Steps**, click **+** next to **SMS OTP View** and connect the Send SMS OTP success path to this new View node.
5. Add a **Verify SMS OTP** executor and connect the OTP View's output to it.
6. Add an **END** node and connect the Verify SMS OTP success path to it.
7. Click the settings icon on the END node:
   - Turn on **Activate Account**.
   - Leave **Auto-Login** off if you want the user to sign in separately after verification.
8. Connect all failure paths to appropriate error View nodes.
9. Click **Save**.

## Example — Registration with Consent

For applications that require users to accept terms of service or a privacy policy, add a **User Consent** executor to the flow.

### Where to Add It

Insert the **User Consent** executor after the registration View node and before the **Provisioning** executor:

```
START → Registration Form (View) → User Consent (Executor) → Provisioning (Executor) → END
```

### How to Set It Up

1. Build the basic registration flow (View + Provisioning + END) as described in the first example.
2. Expand **Executors** in the left panel and add **User Consent** to the canvas.
3. Disconnect the link between the View node and the Provisioning executor.
4. Connect the View node to the **User Consent** executor, then connect User Consent's success path to **Provisioning**.
5. Connect the User Consent failure path to a View node that explains consent is required.
6. Click **Save**.

## Enable Registration on an Application

After saving your registration flow, enable it on the application:

1. Go to **Resources** > **Applications**.
2. Open the application.
3. Navigate to the **Registration Flow** tab.
4. Turn on **Allow Self-Registration**.
5. Select your registration flow from the dropdown.
6. Click **Save**.

A **Sign Up** link automatically appears on the application's login page once registration is enabled.

## Design Considerations

- **Activate on completion or after verification.** Turn off **Activate Account** on the END node and add a verification step before END when you want to confirm the user's identity before activation.
- **Auto-login after registration.** When **Auto-Login** is on, add an **Auth Assertion Generator** executor before the END node and connect it between the final verification step and END.
- **Error paths on every executor.** Always connect each executor's red dot to an error View so users receive clear feedback if any step fails.

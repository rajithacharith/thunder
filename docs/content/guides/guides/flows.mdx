# Flows

Flows are node-based graphs that control user interaction and backend processing during authentication and registration. Each flow consists of nodes (steps), edges (connections), and actions (user-triggered navigation).

## User Disambiguation

When an identifying attribute (such as `email` or `username`) matches more than one user, the **IdentifyingExecutor** in `resolve` mode enables interactive disambiguation. Instead of failing with an error, the flow presents the user with selectable options derived from the matching candidates, allowing them to narrow down to a single identity.

### When to Use Disambiguation

Use the `resolve` mode when your user base allows multiple users to share the same identifying attribute. Common scenarios include:

- Multiple users with the same `email` across different organization units
- Shared `username` values across different user types (e.g., `employee` vs `contractor`)
- Any deployment where a single identifying attribute does not guarantee uniqueness

### How It Works

The disambiguation flow follows a multi-step process:

1. **Initial identification** — The `IdentifyingExecutor` in `resolve` mode searches for all users matching the provided input (e.g., `email`).
2. **Unique match** — If exactly one user matches, the executor completes immediately with the resolved `userID`.
3. **Ambiguous match** — If multiple users match, the executor:
   - Stores the candidate users in `RuntimeData` for later calls.
   - Returns `ExecUserInputRequired` with `ForwardedData` containing distinct attribute values from the candidates (e.g., available `userType` or `department` values).
   - The next prompt node can use these forwarded options to render a dropdown for the user to select from.
4. **Refinement** — On the next submission, the executor filters the stored candidates in-memory using the newly provided attribute. This repeats until a single user remains or no candidates match.
5. **Downstream authentication** — Once resolved, the `userID` is stored in `RuntimeData`. Downstream executors like `BasicAuthExecutor` detect the pre-resolved `userID` and skip the username input, requiring only the password.

### Configuring a Disambiguation Flow

To enable disambiguation, set the `IdentifyingExecutor` mode to `resolve` in the flow definition:

```json
{
  "name": "Disambiguation Authentication Flow",
  "handle": "disambig-auth-flow",
  "flowType": "AUTHENTICATION",
  "nodes": [
    {
      "id": "start",
      "type": "START",
      "onSuccess": "prompt_email"
    },
    {
      "id": "prompt_email",
      "type": "PROMPT",
      "inputs": [
        {
          "identifier": "email",
          "type": "EMAIL_INPUT",
          "required": true
        }
      ],
      "actions": [
        {
          "ref": "action_001",
          "nextNode": "identify_user"
        }
      ]
    },
    {
      "id": "identify_user",
      "type": "TASK_EXECUTION",
      "executor": {
        "name": "IdentifyingExecutor",
        "mode": "resolve"
      },
      "onSuccess": "basic_auth",
      "onIncomplete": "prompt_disambiguate"
    },
    {
      "id": "prompt_disambiguate",
      "type": "PROMPT",
      "inputs": [
        {
          "identifier": "userType",
          "type": "DROPDOWN",
          "required": true
        }
      ],
      "actions": [
        {
          "ref": "action_001",
          "nextNode": "identify_user"
        }
      ]
    },
    {
      "id": "basic_auth",
      "type": "TASK_EXECUTION",
      "executor": {
        "name": "BasicAuthExecutor"
      },
      "onSuccess": "auth_assert"
    },
    {
      "id": "auth_assert",
      "type": "TASK_EXECUTION",
      "executor": {
        "name": "AuthAssertExecutor"
      },
      "onSuccess": "end"
    },
    {
      "id": "end",
      "type": "END"
    }
  ]
}
```

### Executor Modes for IdentifyingExecutor

| Mode | Description |
|------|-------------|
| `identify` | Default mode. Returns a single user or fails if multiple users match the provided attributes. |
| `resolve` | Disambiguation mode. When multiple users match, stores candidates and returns selectable options via `ForwardedData` to enable interactive resolution. |

### Runtime Data Behavior

The disambiguation process uses `RuntimeData` to maintain state across multiple interactions:

| Key | Description |
|-----|-------------|
| `userID` | Set when a unique user is resolved. Consumed by downstream executors (e.g., `BasicAuthExecutor`). |
| `candidateUsers` | Internal key storing serialized candidate users between disambiguation rounds. Automatically managed by the executor. |

### ForwardedData and Dynamic Inputs

When the executor returns an ambiguous result, it populates `ForwardedData` with disambiguation options:

```json
{
  "inputs": [
    {
      "identifier": "userType",
      "options": ["employee", "contractor"]
    },
    {
      "identifier": "department",
      "options": ["engineering", "marketing"]
    }
  ]
}
```

The executor only includes attributes that have more than one distinct value across candidates, as single-value attributes do not help with disambiguation. The downstream prompt node can use these options to render dropdown inputs.

### Pre-Resolved User in BasicAuthExecutor

When a `userID` is already present in `RuntimeData` (set by the `IdentifyingExecutor`), the `BasicAuthExecutor` skips the username input and only requires the password. This enables a seamless transition from disambiguation to authentication without re-entering identifying attributes.

## Federated Authentication with Multi-OU Disambiguation

When users authenticate through a federated identity provider (Google, GitHub, etc.) and exist in multiple organization units, the flow must determine which local account to use. The `FederatedAuthResolverExecutor` handles this by authenticating the resolved user after the `IdentifyingExecutor` narrows down the candidates.

### Why a Dedicated Executor

In federated flows, the identity provider has already verified the user's identity. The remaining question is which local account to associate with. The `IdentifyingExecutor` (a utility executor) handles the disambiguation UI but cannot mark the user as authenticated because the flow engine restricts that to authentication-type executors. The `FederatedAuthResolverExecutor` bridges this gap: it reads the stored candidates, filters by the user's selection, and sets the authenticated user.

### Flow Structure

A typical multi-OU federated authentication flow:

```
Google/GitHub Auth
  |
  v
IdentifyingExecutor (resolve mode)
  |-- 1 match --> Authorization Check
  |-- 0 matches --> Create Organization
  |-- multiple --> Disambiguation Prompt
                      |
                      v
                FederatedAuthResolverExecutor
                  |-- 1 match --> Authorization Check
                  |-- still ambiguous --> Disambiguation Prompt (chained)
```

### Configuring the Flow

The federated auth node must set `allowAuthenticationWithoutLocalUser: true` so that ambiguous users can proceed to the `IdentifyingExecutor` for resolution:

```yaml
- id: google_auth
  type: TASK_EXECUTION
  properties:
    allowAuthenticationWithoutLocalUser: true
    idpId: google-idp
  executor:
    name: GoogleOIDCAuthExecutor
    inputs:
      - identifier: code
        type: TEXT_INPUT
        required: true
  onSuccess: resolve_user

- id: resolve_user
  type: TASK_EXECUTION
  executor:
    name: IdentifyingExecutor
    mode: resolve
    inputs:
      - identifier: sub
        type: TEXT_INPUT
        required: true
  onSuccess: authorization_check
  onFailure: create_organization
  onIncomplete: prompt_disambiguate

- id: prompt_disambiguate
  type: PROMPT
  # Prompt components for OU or attribute selection
  prompts:
    - inputs:
        - identifier: ouHandle
          type: SELECT
          required: true
      action:
        ref: action_select
        nextNode: resolve_auth

- id: resolve_auth
  type: TASK_EXECUTION
  executor:
    name: FederatedAuthResolverExecutor
    inputs:
      - identifier: ouHandle
        type: TEXT_INPUT
        required: true
  onSuccess: authorization_check
  onFailure: prompt_disambiguate
  onIncomplete: prompt_disambiguate
```

### FederatedAuthResolverExecutor

The `FederatedAuthResolverExecutor` is an authentication-type executor that resolves an ambiguous user from stored candidates after federated identity verification.

| Property | Value |
|----------|-------|
| **Executor name** | `FederatedAuthResolverExecutor` |
| **Executor type** | Authentication |
| **Purpose** | Authenticate a user resolved from disambiguation after federated auth |

**Behavior:**
- Reads stored candidate users from `RuntimeData` (set by the `IdentifyingExecutor` during disambiguation).
- Filters candidates against all provided user inputs using generic attribute matching (for example, `ouHandle`, `userType`, or any distinguishing attribute).
- If exactly one candidate remains, sets `AuthenticatedUser` with the resolved user's details.
- If multiple candidates still match, returns `ExecUserInputRequired` with remaining disambiguation options, supporting multi-round chaining.
- If no candidates match, returns failure.

:::warning
Only use the `FederatedAuthResolverExecutor` after a federated authentication step (Google, GitHub, OIDC, etc.) has verified the user's identity. Do not use it in credential-based flows where the user has not yet proven account ownership.
:::

### Disambiguation Input Types

The flow designer controls how disambiguation options are presented. The backend works the same regardless of the UI component:

| Component | Behavior | Security |
|-----------|----------|----------|
| `SELECT` (dropdown) | Shows available options from `ForwardedData` | Reveals which organizations or types the user belongs to |
| `TEXT_INPUT` (free text) | User types the value manually | No disclosure of membership |

Choose `TEXT_INPUT` if OU membership disclosure is a concern for your deployment.

### Runtime Data Keys

| Key | Set by | Description |
|-----|--------|-------------|
| `sub` | OAuthExecutor | The federated subject identifier, used by the IdentifyingExecutor to search for candidates |
| `userAmbiguous` | OAuthExecutor | Set to `"true"` when multiple local users match the federated `sub` claim |
| `userEligibleForProvisioning` | OAuthExecutor | Set to `"true"` only for genuinely new users (no local account exists) |
| `candidateUsers` | IdentifyingExecutor | Serialized candidate users stored between disambiguation rounds |

### Cross-OU Provisioning

When inviting a user who already exists in another organization unit, the federated auth node in the invite flow must enable cross-OU provisioning:

```yaml
- id: google_auth
  type: TASK_EXECUTION
  properties:
    allowAuthenticationWithoutLocalUser: true
    allowRegistrationWithExistingUser: true
    allowCrossOUProvisioning: true
    idpId: google-idp
  executor:
    name: GoogleOIDCAuthExecutor
    inputs:
      - identifier: code
        type: TEXT_INPUT
        required: true
  onSuccess: provisioning
```

| Property | Description |
|----------|-------------|
| `allowCrossOUProvisioning` | When `true`, allows an existing user to be provisioned into a different target OU. Used together with `allowRegistrationWithExistingUser` in invite and onboarding flows. |

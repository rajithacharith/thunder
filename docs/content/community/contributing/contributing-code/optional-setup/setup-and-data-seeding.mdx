# Setup and Data Seeding

<ProductName /> seeds its default resources through an in-process `bootstrap` subcommand, depending on how the system is initialized.

| Command | Use Case | Seeded Data |
|---------|----------|-------------|
| `make run` | Source development | Default resources |
| `./setup.sh` | Deployment / release initialization (seed only) | Default resources |
| `./start.sh --bootstrap-and-serve` | Local/dev: seed once, then serve | Default resources |

All paths run the same in-process `bootstrap` subcommand, which creates the core resources through the service layer (no temporary server, no security bypass):

- Default Organization Unit
- Default User Type (Person) and Agent Type
- Admin user (credentials default to `admin` / `admin`; configurable via `ADMIN_USERNAME` / `ADMIN_PASSWORD` environment variables)
- System resource server with system resources, actions, and permissions
- Administrators group, Administrator role, and the role assignment
- `Console` application
- Default flows, themes, and translations

## Development Seeding (`make run`)

For development environments, `make run` runs `go run … bootstrap` against the local SQLite databases before starting the server.

## Full Bootstrap (`./setup.sh`)

For release artifacts and deployment environments, `./setup.sh` performs a one-time initialization. It delegates to `./start.sh --bootstrap`, which runs the compiled binary's in-process `bootstrap` one-shot (creating the resources above, then exiting) and starts the bundled consent server for the duration of the run. The bootstrap reads the resource definitions from `bootstrap/`.

## Seed and Serve in One Step (`./start.sh --bootstrap-and-serve`)

For local and development runs of a built distribution, `./start.sh --bootstrap-and-serve` runs the in-process `bootstrap` one-shot, then starts the long-running server. The bootstrap is idempotent (it skips resources that already exist), and the server reuses the consent server it started. `./start.sh --bootstrap` is the seed-only variant (used internally by `setup.sh`).

**In short:**
- Use `make run` when running <ProductName /> from source.
- Use `./setup.sh` when initializing a fresh runtime environment as a separate one-time step (e.g. a deployment init job), then `./start.sh` to serve.
- Use `./start.sh --bootstrap-and-serve` to seed and serve a built distribution in a single command (local/dev).

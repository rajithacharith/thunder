# Security Token Service

A Security Token Service (STS) is the trust engine that lets applications exchange, transform, and broker identities across service boundaries. <ProductName /> provides STS capabilities based on OAuth 2.0 Token Exchange ([RFC 8693](https://www.rfc-editor.org/rfc/rfc8693)) — so you can issue audience-specific tokens, reduce scope for least-privilege access, broker identities across identity providers, and secure AI agent interactions without building token transformation infrastructure from scratch.

## When to Use This Pattern

Use this pattern if your platform follows a service-oriented or microservices architecture and you need a trusted authority to issue, transform, and validate tokens across service boundaries. Specifically, choose STS when you have:

- Services that call other services and need audience-scoped tokens at each hop
- APIs that accept tokens from multiple upstream callers or external identity providers
- Workflows where a service must act on behalf of a user with explicit delegation
- AI agents that need short-lived, scoped credentials before performing actions
- A requirement to enforce least-privilege access at each service boundary

## How It Works

In this model:

- Services exchange tokens at the `POST /oauth2/token` endpoint using the `urn:ietf:params:oauth:grant-type:token-exchange` grant type.
- <ProductName /> validates the subject token and checks that the requesting application has the token exchange grant type authorized.
- The issued token carries the correct audience binding, reduced scope, and — when provided — actor context from the `actor_token`.
- Downstream services validate tokens using <ProductName />'s JWKS endpoint and standard JWT claims (`iss`, `sub`, `aud`, `exp`, `iat`).
- Scope reduction is enforced automatically: <ProductName /> grants only the intersection of requested scopes and scopes present in the subject token.
